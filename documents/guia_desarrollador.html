Guía de Inicio Rápido para Desarrolladores

Pasos esenciales para la instalación y ejecución del ecosistema de microservicios.

0. Configuración Global (Requisitos)

Requisito 1: Contenedores

Es **obligatorio** usar Docker y Docker Compose para orquestar la DB y los servicios.

  • Instalar Docker Desktop.

Requisito 2: Node.js

Necesario para el **Products Service** y herramientas de Frontend (Angular).

  • Instalar **Node.js (v18+)** y **npm/yarn**.

Requisito 3: Python

Necesario para el **Inventory Service**.

  • Instalar **Python 3.9+**.
  • Configurar el alias python3 o python.

Archivo de Configuración Global

Antes de iniciar, cree el archivo de variables de entorno en la raíz del proyecto (use el archivo .env.example como plantilla):

cp .env.example .env

Revise y edite el archivo .env para configurar la clave PRODUCTS_API_KEY y las credenciales de MySQL (ver **Guía de Variables de Entorno** para detalles).

1. Products-Service (Node.js/Express)

Módulo CRUD central. Se encuentra en: backend/products-service/

Estructura Clave (Clean Architecture)

  • models/: **Patrón Repositorio** (consultas SQL).
  • logic/: **Patrón de Servicio** (lógica de negocio y validaciones).
  • middleware/: Contiene authApiKey.js (Autenticación).
  • routes/: Mapeo de URLs (`/api/v1/productos`).

Instalación y Ejecución Local

# 1. Instalar dependencias
cd backend/products-service
npm install

# 2. Ejecutar el servicio
npm run dev

# Servicio disponible en: http://localhost:8001
# Documentación API: http://localhost:8001/swagger-products/

Ejecución de Pruebas (Jest)

Objetivo: **80% de Cobertura**. El reporte HTML se genera en test_results/.

npm test

2. Inventory-Service (Python/Flask)

Módulo de stock y lógica compleja. Se encuentra en: backend/inventory-service/

Estructura Clave (Clean Architecture)

  • logic/: Contiene el **Cliente HTTP Resiliente** para Products Service.
  • middleware/: Contiene error_handler.py (Logging Estructurado).
  • exceptions/: Excepciones tipadas (para errores JSON API).

Instalación y Ejecución Local

# 1. Crear y activar Entorno Virtual (obligatorio)
cd backend/inventory-service
python3 -m venv venv
source venv/bin/activate  # Para Linux/macOS

# 2. Instalar dependencias
pip install -r requirements.txt

# 3. Ejecutar el servidor
export FLASK_ENV=development 
python app.py

# Servicio disponible en: http://localhost:8000

Ejecución de Pruebas (Pytest)

Objetivo: **80% de Cobertura**. El reporte HTML se genera en test_results/.

# Con (venv) activo
pytest --cov=. --cov-report html:test_results/inventory-coverage

3. Frontend (Angular)

Aplicación web para consumir las APIs. Se encuentra en: frontend/products-manage/

Requisitos y Estructura

  • Instalar el Angular CLI: npm install -g @angular/cli
  • El enfoque de pruebas (30-40%) se centra en **Servicios** e **Interceptors** (manejo de errores 401).

Instalación y Ejecución Local

# 1. Instalar dependencias
cd frontend/products-manage
npm install

# 2. Ejecutar el servidor de desarrollo
ng serve

# Aplicación disponible en: http://localhost:4200/

Ejecución de Pruebas (Karma/Jasmine)

Para ejecutar las pruebas unitarias:

ng test

4. Futuras Mejoras y Próximos Pasos

Esta sección detalla las propuestas de mejora y ampliación del proyecto, superando los requisitos iniciales del nivel Líder Técnico:

1. Estrategia de Comunicación Asíncrona (Message Bus)

  • **Objetivo:** Reducir el acoplamiento y mejorar la escalabilidad y resiliencia del *backend*.
  • **Propuesta:** Reemplazar la emisión de logs básicos por la implementación de un **Bus de Mensajes / Cola de Eventos** (ej. RabbitMQ o Kafka). Esto permitirá que las actualizaciones de inventario se notifiquen mediante eventos asíncronos (inventory.updated), desacoplando completamente a los consumidores (Notificaciones, Analytics) del servicio de Inventario.

2. Cobertura Total de Pruebas de Frontend (E2E)

  • **Objetivo:** Complementar las pruebas unitarias de Angular con pruebas de validación de experiencia de usuario.
  • **Propuesta:** Implementar pruebas **End-to-End (E2E)** utilizando herramientas como **Selenium** o Cypress. Esto garantizará que los flujos críticos del usuario (Listado, Consulta de stock, Actualización de compra) funcionen correctamente desde el navegador, simulando interacciones reales.

3. Automatización de Despliegues (Git Actions)

  • **Objetivo:** Lograr una integración y despliegue continuo (CI/CD).
  • **Propuesta:** Crear un documento y posteriormente los *workflows* de **Git Actions** para automatizar: 1) La ejecución de pruebas unitarias/integración/E2E en cada *push*. 2) La construcción de las imágenes Docker (Products/Inventory/Frontend). 3) El despliegue automático a un entorno de *staging*/producción tras una fusión a la rama principal.

Consulte los documentos de **Arquitectura**, **Seguridad** y **Pruebas** para conocer los detalles de diseño.
